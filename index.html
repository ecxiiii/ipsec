Authentication Header (AH)

IP Protocol: 51

Provides

  • Data integrity via MAC
  • Origin authentication
  • Anti-replay protection
  • Authenticates immutable IP header fields

Header Fields

  • Next Header: Following protocol ID
  • SPI: SA identifier
  • Sequence Number: Anti-replay counter
  • Auth Data: ICV hash

‚ö†Ô∏è Limitations

‚Ä¢ NAT incompatible (authenticates IP header)
‚Ä¢ No confidentiality (cleartext)
‚Ä¢ Less commonly deployed than ESP

ESP (Encapsulating Security Payload)

IP Protocol: 50

Provides

  • Confidentiality (encryption)
  • Optional authentication & integrity
  • Anti-replay protection
  • NAT compatible (doesn't protect outer IP header)

Internet Key Exchange (IKE)

Automates SA Creation & Key Exchange

Primary Functions

  1. Mutual Authentication: PSK, certificates, or public keys
  2. Key Generation: Shared symmetric keys
  3. SA Negotiation: Agree on algorithms & parameters
  4. Key Refresh: Periodic renewal

Two-Phase Structure

Phase 1

‚Ä¢ Secure channel setup
‚Ä¢ IKE SA / ISAKMP tunnel
‚Ä¢ Expensive (DH exchange)

Phase 2

‚Ä¢ IPsec SA negotiation
‚Ä¢ Uses Phase 1 channel
‚Ä¢ Can be repeated

Transport Mode

Protects Payload Only

Characteristics

  • Original IP header exposed
  • Encrypts/authenticates payload only
  • IPsec header between IP header & payload
  • Lower overhead
  • Preserves original MTU

ESP Transport Structure

IP Hdr ESP Hdr üîí Payload Trailer Auth

Use Cases

  • Host-to-host direct
  • Client-to-server VPN
  • Both endpoints support IPsec

Tunnel Mode

Encapsulates Entire Packet

Characteristics

  • Entire packet becomes payload
  • New outer IP header (tunnel endpoints)
  • Original IP header encrypted
  • Higher overhead (dual headers)
  • Reduced MTU
  • Hides internal routing info

ESP Tunnel Structure

New IP ESP üîí Orig IP + Data Trailer Auth

Use Cases

  • Site-to-site VPN
  • Security gateways
  • Remote access VPN
  • Traffic analysis protection

Transport vs Tunnel Mode

IKE Phase 2

Quick Mode - IPsec SA Negotiation

Objectives

  1. Negotiate protocol (AH/ESP) & mode
  2. Select encryption & auth algorithms
  3. Derive session keys from Phase 1
  4. Establish SAs (min 2 for bidirectional)

PFS (Perfect Forward Secrecy)

‚Ä¢ Optional additional DH exchange

IKEv1 vs IKEv2

IKEv1 (RFC 2409)

  • Main Mode / Aggressive Mode
  • Quick Mode (Phase 2)
  • Complex exchanges
  • Legacy, widely deployed

IKEv2 (RFC 4306)

  • Simplified design
  • Fewer messages
  • Built-in NAT-T
  • Explicit ACKs
  • Better remote access VPN
  • DoS resistant
  • Single exchange type

‚úì Recommended for new deployments

Security Best Practices

Algorithms

  • AES-256 or ChaCha20
  • SHA-256 or stronger
  • DH Group 14+ (2048-bit)
  • Enable PFS
  • Avoid 3DES, DES, SHA-1, MD5

Key Management

  • Use IKE (dynamic)
  • Regular key refresh
  • Certificates > PSK
  • Strong passphrases

Policy Config

  • Restrictive SPD (least-privilege)
  • Audit regularly
  • ESP encrypt + auth
  • Anti-replay ON

Operations

  • Monitor logs
  • Apply security patches
  • Test failover
  • Document configs
